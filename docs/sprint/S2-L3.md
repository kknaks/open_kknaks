## 구현 결정사항

1. **before 예외 시 해당 MW의 after 미호출**: `called_middlewares.append(mw)`는 before 성공 후에만 실행. before에서 예외를 일으킨 MW 자신은 after를 받지 않음. Dramatiq과 미세한 차이가 있으나 의도적 결정 — before가 실패한 MW는 초기화가 안 된 상태이므로 after에서 정리할 것이 없음.

2. **BudgetExceededError는 BillingError의 하위 클래스**: CostMiddleware의 예산 초과가 RetriesMiddleware의 `no_retry_on`에 자동 포함됨. 예산 초과 시 재시도 방지.

3. **Worker.stop() 이중 호출 방어**: `_stopping` 플래그로 SIGTERM+SIGINT 연속 호출 방어.

4. **Worker.stop() requeue**: force terminate 시 active task ID를 수집하여 broker.requeue() 호출. 태스크 유실 방지.

5. **RateLimitMiddleware**: preemptive(before, 분당 제한) + reactive(after, 429 감속/성공 복구) 듀얼 방식. asyncio 단일 스레드이므로 별도 Lock 불필요.

6. **task.max_retries=0 의미**: 현재 기본값 0은 "middleware 기본값 사용"으로 해석. 명시적 "재시도 안 함"은 RetriesMiddleware를 미들웨어 체인에서 제거하거나 no_retry_on을 확장하는 방식으로 처리. (WARN으로 기록, 추후 Task 필드를 Optional[int]로 변경 검토)

## 발견된 이슈

1. (WARN) `task.max_retries` 기본값 0이 "사용 안 함"인지 "기본값 사용"인지 모호. 추후 None으로 변경 고려.
2. (WARN) `subscribe_chunks` 타임아웃 미구현 — S3에서 Client 구현 시 함께 해결.
