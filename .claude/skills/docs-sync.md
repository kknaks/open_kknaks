# /docs-sync 스킬

코드 변경 사항이 설계 문서와 일치하는지 검사하고, 불일치하면 문서를 업데이트한다.
문서 전체 목록은 `docs/INDEX.md` 참조.

## 사용법

- `/docs-sync` — 최근 변경된 파일 기준으로 전체 검사
- `/docs-sync {파일경로}` — 특정 파일 변경에 대해 검사

## 검사 대상

### 설계 문서 (코드 ↔ 문서 일치 검사)

| 문서 | 검사 포인트 |
|---|---|
| `docs/ARCHITECTURE_V2.md` | 클래스 시그니처, 메서드 인터페이스, 데이터 흐름, 코드 예시 |
| `docs/PRD.md` | API 인터페이스, Task/Config 모델 필드, CLI 플래그 매핑 |
| `docs/PLAN.md` | 스프린트 작업 상태, 설계 결정 테이블, 의존성 버전 |
| `CLAUDE.md` | 패키지 구조, 의존성 버전, 개발 명령어 |

### 스프린트 문서 (구현 기록 반영)

| 문서 | 검사 포인트 |
|---|---|
| `docs/sprint/S{n}-L{n}.md` | 구현 결정사항, 발견 이슈, 인터페이스 변경 이력 |

### 문서 인덱스

| 문서 | 검사 포인트 |
|---|---|
| `docs/INDEX.md` | 새 문서 추가 시 인덱스에 등록되어 있는지 |

## 실행 흐름

1. `git diff --name-only` 또는 지정된 파일로 변경 목록 파악
2. 변경된 소스 파일을 읽고 핵심 변경 사항 추출:
   - 클래스/함수 시그니처 변경
   - 필드 추가/삭제/타입 변경
   - 새 모듈/파일 추가
   - 의존성 변경
   - 설계 패턴 변경
3. `docs/INDEX.md`를 읽어 관련 문서 목록 파악
4. 각 문서를 읽고 변경 사항과 불일치하는 부분 탐색:
   - 코드 예시가 실제 코드와 다른가
   - 필드 목록이 실제 모델과 다른가
   - 인터페이스 설명이 실제 구현과 다른가
   - 스프린트 완료 기준이 달라졌는가
   - 새 문서가 INDEX.md에 누락되었는가
5. 불일치 발견 시:
   - 변경 내용과 위치를 유저에게 보여준다
   - 유저 확인 후 문서를 수정한다
6. 전부 일치하면 한 줄로 "문서 동기화 완료" 출력

## 주의사항

- 문서의 설계 의도를 바꾸지 않는다. 코드와 불일치하는 **사실관계**만 수정한다.
- 설계 결정이 바뀐 경우 (예: 새 필드 추가 이유), 유저에게 설명을 요청한다.
- PLAN.md의 체크리스트(`- [ ]` / `- [x]`)는 실제 구현 상태에 맞게 업데이트한다.
- 새 문서가 생기면 `docs/INDEX.md`에 등록한다.
