# 예시: v2.0.0 BREAKING release

이 스킬을 처음 만들 때 함께 진행한 실제 사례.

## 변경 내용 (요약)

- `TaskResult.output` 제거 → `TaskResult.result` (최종 텍스트) + `TaskResult.stream` (raw 합본)으로 분리.
- stream-json 파서가 `result` 메시지의 텍스트를 silently drop하던 버그 수정.
- 결과: 한글 partial chunk가 `\n`으로 분절되던 문제(`"이러\n한"`) 해결.

## 왜 major bump였나

`TaskResult.output` 필드 자체가 사라졌으므로 기존 사용자 코드가 깨진다. SemVer 규칙상 major(2.0.0)가 맞다.
patch(1.1.1)로 가자는 의견이 있었지만, "필드 의미가 바뀌는 것"은 patch가 보장하는 "동작 무변화"와 양립할 수 없으므로 거부.

## 진행 순서

### 1. 코드/테스트/문서 변경 + 커밋

```bash
# 변경한 파일들:
#   open_kknaks/task.py                       (TaskResult 스키마)
#   open_kknaks/worker/stream_parser.py       (source 필드, cost+text 둘 다 emit)
#   open_kknaks/worker/executor.py            (result_text + stream_parts 분리)
#   open_kknaks/worker/worker.py              (task.result = result.result)
#   tests/unit/test_stream_parser.py          (회귀 테스트 추가)
#   tests/unit/test_task.py                   (TaskResult 새 필드 검증)
#   tests/integration/test_middleware.py      (output= → result=)
#   pyproject.toml                            (ruff extend-exclude _version.py)
#   CHANGELOG.md                              (신규)
#   scripts/smoke_v2.py                       (실 CLI 스모크)

git add -A
git commit -m "feat!: split TaskResult.output into .result and .stream

BREAKING CHANGE: ...
"
```

### 2. CHANGELOG.md 작성

`## [2.0.0] — 2026-04-30` 섹션에 Breaking / Fixed / Added 작성. 마이그레이션 가이드 포함:

```markdown
# before (v1.x)
text = task_result.output

# after (v2.0)
text = task_result.result
raw  = task_result.stream
```

### 3. 실 CLI smoke test

`scripts/smoke_v2.py` 작성 후:

```bash
uv run python scripts/smoke_v2.py
```

확인:
- `task.result` = 깔끔한 한글 텍스트, `\n` 없음 ✓
- `task.stream` = `"이러\n한 트렌드"`처럼 분절 흔적 보임 ✓
- `on_chunk text count > 0` ✓

이 스모크가 없었다면 "파서가 가정대로 동작하는지" 단위 테스트만으로는 알 수 없었다.

### 4. /release 스킬로 배포

```
/release 2.0.0
```

내부 동작:
1. `preflight.sh` — clean tree, on main, 동기화, ruff/mypy/pytest 모두 통과
2. CHANGELOG에 `## [2.0.0]` 있는지 검사 → 있음
3. `git push origin main`
4. `git tag -a v2.0.0 -m "Release v2.0.0"`
5. `git push origin v2.0.0`
6. GitHub Actions `release.yml` 자동 트리거
7. `gh run list` 출력으로 모니터링 시작점 안내

### 5. CI 모니터링

```bash
gh run watch
```

Actions가 진행하는 단계:
- Lint (ruff check + format)
- Type check (mypy --strict)
- Test (pytest, e2e 제외)
- Build (uv build → dist/ 에 sdist + wheel)
- Publish (uv publish, PYPI_API_TOKEN secret 사용)
- GitHub Release 생성 + dist/ 자산 업로드

성공하면 PyPI에서 `2.0.0` 확인 가능: https://pypi.org/project/open-kknaks/2.0.0/

## 교훈

- **major bump의 가치**: SemVer 신호가 명확하면 사용자가 마이그레이션을 준비할 수 있다. CHANGELOG 마이그레이션 스니펫 필수.
- **silent drop 버그는 단위 테스트만으로 못 잡는다**: "버그가 있는 줄 모르는" 코드는 그 동작을 검증하는 테스트가 있어 통과하기 마련. 실 CLI smoke가 안전망.
- **/release 스킬화의 효과**: "어떤 명령을 어떤 순서로?"를 매번 떠올리지 않아도 된다. 실수의 80%는 순서 혼동에서 옴.
